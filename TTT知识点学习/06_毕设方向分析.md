## 一、研究背景与选题概述

### 1.1 TTT技术背景

TTT（Test-Time Training，测试时训练）是一种新兴的序列建模架构，其核心创新在于**将隐藏状态从固定向量转变为可学习的模型**。与传统的Transformer和Mamba相比，TTT在长上下文场景下具有显著优势：

- **推理延迟恒定**：不随序列长度增长（TTT约25ms vs Transformer在128K长度时达1.8s）
- **长上下文表现优异**：128K大海捞针任务准确率79.8%，远超Transformer的58.4%
- **自适应学习能力**：在推理过程中动态适应输入分布

根据参考文档《06_未来趋势预测.md》，TTT正处于从原型发布（2025-2026）向产品化阶段（2027-2029）过渡的关键时期，此时参与研究具有较好的前沿性和实践价值。

### 1.2 本科生能力画像

基于你提供的背景信息，具备以下能力基础：

- **Transformer基础**：理解注意力机制、前向传播、反向传播等核心概念
- **模型训练经验**：从0到1训练过28M参数的小模型
- **硬件资源**：TTT-E2E环境（JAX框架，9块RTX 3090）
- **SSH访问**：学校服务器10.1.36.65，可远程训练

这些条件使得开展TTT相关研究成为可能。28M模型训练经验表明已经掌握深度学习训练的基本流程，具备进一步探索TTT的起点。

## 三、详细研究方向与课题推荐

### 3.1 架构改进方向

#### 课题1：TTT-Linear与Transformer混合架构设计

**研究问题定义**

如何在保留Transformer强大表达能力的同时，引入TTT的线性复杂度优势？具体而言，设计一种混合架构，使模型能够在不同层次灵活选择使用标准注意力或TTT层。

**研究问题细化**

- 问题1：混合架构中TTT层与注意力层的最优配比是多少？
- 问题2：如何设计路由机制，使模型自适应选择使用哪种层？
- 问题3：混合架构在长序列任务上的性能边界在哪里？

**实施路径**

```
阶段1：文献调研（第1-2周）
├── 精读TTT论文、Titans论文
├── 梳理混合架构相关工作（Hybrid Mamba-Transformer等）
└── 分析现有混合方案的优缺点

阶段2：架构设计与实现（第3-5周）
├── 设计混合层配置（交替式、级联式、并行式）
├── 实现动态路由机制（基于输入复杂度/任务类型）
└── 在JAX框架下实现混合模块

阶段3：实验验证（第6-10周）
├── 短序列基线对比（2K-8K）
├── 长序列消融实验（16K-128K）
├── 不同路由策略对比
└── 效率分析（延迟、内存、吞吐量）

阶段4：论文撰写（第11-12周）
├── 结果分析与可视化
├── 撰写技术报告
└── 准备答辩材料
```

**所需技术栈**

- 框架：JAX/Flax（已有TTT-E2E环境）
- 工具：PyTorch对比实验、wandb实验管理
- 数学：线性代数、注意力机制

**预期工作量**：12周

**难度评估**：4星（创新性课题）

**推荐理由**：
- 混合架构是未来趋势（见《06_未来趋势预测.md》2.1.1节）
- 有明确的改进空间和清晰的评估指标
- 符合"从0到1训练过28M模型"的能力基础

#### 课题2：TTT-MLP的改进方案研究

**研究问题定义**

TTT-MLP具有比TTT-Linear更强的表达能力（在32K上下文BooksCorpus上困惑度11.3 vs 11.8），但训练稳定性较差（需要更小的内循环学习率0.1 vs 1.0）。如何改进TTT-MLP的训练稳定性，同时保持其表达能力优势？

**研究问题细化**

- 问题1：TTT-MLP训练不稳定性的根源是什么？
- 问题2：如何设计更适合TTT-MLP的优化器或学习率调度？
- 问题3：能否引入正则化或梯度裁剪策略改善稳定性？

**实施路径**

```
阶段1：稳定性分析（第1-2周）
├── 复现TTT-MLP训练过程
├── 绘制损失曲线，分析不稳定的阶段
├── 梯度分布统计分析

阶段2：改进方案设计（第3-6周）
├── 设计1：自适应学习率（类似RAdam）
├── 设计2：梯度裁剪策略
├── 设计3：混合优化器（内循环Adam + 外循环SGD）
└── 设计4：引入LayerNorm到隐藏状态更新

阶段3：实验验证（第7-10周）
├── 不同改进方案对比
├── 超参数敏感性分析
├── 与TTT-Linear、Transformer的性能对比
└── 消融实验确定最优配置

阶段4：总结与撰写（第11-12周）
```

**所需技术栈**

- 框架：JAX（TTT-E2E环境）
- 工具：梯度分析工具、实验管理系统
- 理论：优化理论基础

**预期工作量**：12周

**难度评估**：3星半（改进型课题）

**推荐理由**：
- 问题定义清晰，有明确的改进目标
- 有大量消融实验空间，适合探索性研究
- 属于"改进型"而非"创新型"，对本科生更友好

#### 课题3：基于Gating机制的TTT层设计

**研究问题定义**

参考Titans架构中的MAG（Memory as Gating）思想，设计一种带门控的TTT层，使模型能够动态决定从当前输入中学习多少内容，从而避免过度适应噪声。

**研究问题细化**

- 问题1：门控机制如何设计？基于输入还是基于隐藏状态？
- 问题2：门控的开放程度如何影响模型的记忆能力？
- 问题3：如何在效率和性能之间取得平衡？

**实施路径**

```
阶段1：基础调研（第1-2周）
├── 精读Titans论文MAG部分
├── 调研LSTM门控机制
└── 分析TTT中的"过度学习"问题

阶段2：门控设计（第3-5周）
├── 方案1：输入驱动的标量门控
├── 方案2：隐藏状态驱动的向量门控
├── 方案3：可学习的门控参数
└── 确定门控计算形式

阶段3：实验验证（第6-10周）
├── 不同门控方案的对比
├── 噪声输入下的泛化实验
├── 长序列记忆保持实验
└── 与标准TTT-Linear、TTT-MLP对比

阶段4：总结撰写（第11-12周）
```

**所需技术栈**

- 框架：JAX/Flax
- 实验：标准LLM评估流程

**预期工作量**：12周

**难度评估**：3星半

**推荐理由**：
- 有明确的创新点（门控机制）
- 实验结论可直接对比验证
- 与已有研究（Titans）有明确关联

### 3.2 效率优化方向

#### 课题4：TTT量化推理方法研究

**研究问题定义**

TTT在推理时需要持续更新隐藏状态，这对量化推理提出了独特挑战。如何设计适合TTT的量化方案，在保持性能的同时降低推理成本？

**研究问题细化**

- 问题1：隐藏状态的量化精度要求是什么？
- 问题2：梯度计算对量化误差的敏感度如何？
- 问题3：动态量化还是静态量化更适合TTT？

**实施路径**

```
阶段1：基础调研（第1-2周）
├── 调研LLM量化方法（GPTQ、AWQ、QAT）
├── 分析TTT隐藏状态的数值分布
└── 设计量化实验基线

阶段2：量化方案设计（第3-5周）
├── 方案1：隐藏状态后训练量化（PTQ）
├── 方案2：训练时量化（QAT）
├── 方案3：混合精度（关键层高精度）
└── 设计量化-反量化流程

阶段3：实验验证（第6-9周）
├── 不同精度（FP16/INT8/INT4）下的困惑度
├── 长序列任务性能衰减分析
├── 推理速度提升实测
└── 与标准Transformer量化对比

阶段4：优化与总结（第10-12周）
```

**所需技术栈**

- 框架：JAX（推理）、PyTorch（对比）
- 工具：量化工具、推理基准测试

**预期工作量**：12周

**难度评估**：3星

**推荐理由**：
- 量化是工程落地的关键步骤
- 有明确的性能-效率权衡可探索
- 符合《06_未来趋势预测.md》中"量化+TTT"的趋势预测（2026年INT4 TTT可行）

#### 课题5：TTT Mini-batch优化策略研究

**研究问题定义**

**研究问题细化**

- 问题1：不同batch size对TTT性能的影响规律是什么？
- 问题2：对偶形式相比原始形式的效率提升具体有多少？
- 问题3：能否进一步优化对偶形式的GPU利用率？

**实施路径**

```
阶段1：理论分析（第1-2周）
├── 梳理TTT的原始形式和对偶形式
├── 推导计算复杂度
└── 分析GPU内存访问模式

阶段2：实现优化（第3-5周）
├── 实现标准Mini-batch TTT
├── 实现对偶形式并验证等价性
├── 优化矩阵乘法核函数
└── 实现梯度累积策略

阶段3：性能测试（第6-9周）
├── 不同batch size的性能曲线
├── 对偶形式vs原始形式速度对比
├── 内存占用对比
└── 长序列场景下的效率分析

阶段4：应用探索（第10-12周）
```

**所需技术栈**

- 框架：JAX（已有环境）
- 工具：CUDA性能分析工具

**预期工作量**：10周

**难度评估**：3星

**推荐理由**：
- 问题明确，实验可重复性强
- 效率优化是工程落地的重要环节
- 与你已有的28M模型训练经验直接相关

| 周次 | 任务 |
|------|------|

**常见风险1：环境配置问题**

- 表现：JAX安装失败、GPU无法识别
- 应对：使用Docker镜像、请教有经验的同学

**常见风险2：训练不稳定**

- 表现：损失发散、梯度爆炸
- 应对：降低学习率、添加梯度裁剪、使用TTT-Linear替代TTT-MLP

**常见风险3：实验结果不达预期**

- 表现：复现结果与论文差异较大
- 应对：检查超参数、确认数据处理流程、必要时调整预期
