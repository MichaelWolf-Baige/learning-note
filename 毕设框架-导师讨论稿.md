## 一、题目（候选）

**「基于结构化路径记忆的工具增强型语言智能体决策优化」**

> 英文：*Path Memory: Learning From Decision Trajectories for Efficient Tool-Augmented Agents*

---

## 二、一句话说清楚

> **问题：** Agent 用工具完成任务时，每次都从零开始探索，重复走弯路、重复犯错过。
> **方案：** 把 Agent 每次的成功/失败决策路径结构化存下来，新任务来的时候直接给最短可达路径，让 Agent 少走 60% 的弯路。

---

## 三、核心洞察（为什么现有方案不够好）

| | RAG | Auto-Dreamer / Mem0 | **Path Memory（我们）** |
|---|---|---|---|
| 存什么 | 文档/对话片段 | 压缩后的抽象记忆 | **结构化的决策路径** |
| 失败信息 | 不存 | 压缩时丢掉 | **存为"此路不通"标记** |
| 检索结果 | 相关文本 → Agent 读了再判断 | 抽象知识 → Agent 自己重组 | **可直接执行的最短路径** |

**人类需要遗忘因为大脑容量有限。Agent 不需要遗忘——向量数据库能存几百万条路径，它需要的是精确检索到最短的那条。**

## 四、系统架构

```
Agent 收到新任务 "帮我找 Transformer 最新论文"
         │
         ▼
┌─────────────────────────────────────┐
│        Path Memory MCP Server        │
│                                      │
│  ┌──────────────────────────────┐   │
│  │ 1. Task Matching (ChromaDB)  │   │  ← 把任务编码成向量，找最相似的历史任务
│  └──────────┬───────────────────┘   │
│             ▼                        │
│  ┌──────────────────────────────┐   │
│  │ 2. Path Selection 算法        │   │  ← 最短成功路径优先 + 失败路径过滤
│  └──────────┬───────────────────┘   │
│             ▼                        │
│  ┌──────────────────────────────┐   │
│  │ 3. 返回给 Agent              │   │
│  │  ✅ [search_arxiv → filter → │   │
│  │      fetch top 3]            │   │
│  │  ❌ [search_web] — 噪音多    │   │  ← 失败标记：此路不通
│  │  ❌ [arxiv without filter]   │   │
│  └──────────────────────────────┘   │
│                                      │
│  SQLite (路径结构) + ChromaDB (向量索引) │
└─────────────────────────────────────┘
```

路径数据结构：
```
TaskPath:
  task: "找 Transformer 注意力最新论文"
  success_steps: [search_arxiv(q, filter=2026) → fetch(paper3)]
  failed_attempts: [search_web(q), search_arxiv(q, no filter)]
  avg_steps: 2
  hit_count: 47  (被复用了 47 次)
```

## 五、核心创新（论文 Contribution）

1. **首次定义结构化路径记忆**：将 Agent 记忆从"文本"提升为"可执行决策路径"，信息不损失
2. **失败路径的价值化**：所有现有方案都丢弃失败信息，我们证明保留失败路径能减少重复错误
3. **最短路径检索算法**：传统 RAG 检索"相关内容"，我们检索"最短可达路径"
4. **MCP 标准化接口**：任何 Agent 可插拔使用，已开源实现

## 六、实验设计

### 实验设置
- **Agent**：Qwen2.5:7B（本地 Ollama）+ 5 个搜索工具
- **任务**：50 个信息检索任务（学术搜索、技术调研、新闻追踪）
- **3 组对比**：

| 组 | 记忆方式 | Agent 看到什么 |
|----|---------|---------------|
| No Memory | 无 | 只有当前对话 |
| RAG Memory | 向量检索历史经验 | 5 条相关文本描述 |
| **Path Memory** | 结构化路径检索 | 1 条最短成功路径 + N 条避开标记 |

### 实验流程

```
Phase 1: 数据收集（50 任务 × 从零探索）
  → 构建路径库（成功+失败路径，约 200-300 条）

Phase 2: 记忆检索（同样 50 任务，打乱顺序）
  → 各组利用已建立的记忆库完成任务

Phase 3: 泛化测试（20 个新任务，与训练集相似但不相同）
  → 评估记忆跨任务迁移能力
```

### 核心指标
- **Step Efficiency**（主指标）：完成任务的平均步数
- **Failure Avoidance Rate**：走已知失败路径的比例
- **Cross-Task Transfer**：新任务上的效率提升

### 预期结果（答辩时最有力的图）
```
平均步数对比:
No Memory  ████████  8 步
RAG Memory █████     5 步  (-37%)
Path Memory ██▌      2.5步 (-69%)  ← 一张柱状图说清楚
```

## 附：对可能质疑的准备

> **Q: 和 RAG 有什么区别？不就是另一种 retrieval？**
> A: RAG 返回的是"描述做法的文本"，Agent 需要读了再组织策略。Path Memory 返回的是"可直接执行的步骤序列"。本质区别：一个是素材，一个是答案。

> **Q: 如果新任务和历史任务不完全一样怎么办？**
> A: 这正是 Phase 3 泛化实验要验证的。我们预期相似任务间能正向迁移（相似度 > 0.7 的场景），低于阈值则 fallback 到从零探索。

> **Q: 实验规模是不是太小了（50 个任务）？**
> A: 参考 Evo-Memory (Google DeepMind) 的 benchmark 设计，我们的 50 任务 × 5 维度 = 250 个测试点，覆盖了 5 种记忆使用场景，是可接受的。如果需要可以做更大规模。
